1. Flujo Global del Sistema

Arquitectura general: Frontend → API REST → MySQL → Servicios externos
El sistema sigue una arquitectura de tres capas. El Frontend en React/Vite se comunica con el Backend en Express.js mediante peticiones HTTP (JSON). El backend procesa la lógica de negocio, consulta la base de datos MySQL y, cuando es necesario, llama a servicios externos como Gemini API (OCR, chatbot, becas, bienestar) o SMTP (correos). El middleware JWT protege las rutas privadas y el middleware de roles (RBAC) controla los permisos por perfil.

2. Autenticación — Login

POST /api/auth/login • Middleware: validateInstitutionalEmail
El usuario ingresa su correo y contraseña en el formulario de Login. Se normaliza el email (minúsculas, trim) y se envía al backend. El middleware verifica que el dominio del correo esté en la lista blanca. El controlador busca el usuario en la BD, verifica que esté Activo, compara la contraseña con bcrypt, y si es válida, genera un token JWT firmado con el secret del .env. Se devuelve el token y los datos del usuario. El frontend almacena el token y los datos en localStorage y redirige según el rol. 
sequenceDiagram
    actor U as Usuario
    participant F as Frontend (LoginPage)
    participant B as Backend (auth.js)
    participant DB as MySQL

    U->>F: Ingresa correo + contraseña
    F->>F: normalizeEmail() + validar dominio
    F->>B: POST /auth/login { correo, contrasena }
    B->>B: validateInstitutionalEmail (dominios permitidos)
    B->>DB: SELECT usuarios + roles WHERE correo_institucional = ?
    DB-->>B: Datos del usuario (o vacío)
    B->>B: usuario.estado === 'Activo'?
    B->>B: bcrypt.compare(contrasena, hash)
    B->>B: jwt.sign({ id_usuario, correo, rol }, JWT_SECRET)
    B-->>F: 200 { ok: true, token, usuario: {...} }
    F->>F: localStorage.setItem('sivacad_token', token)
    F->>F: localStorage.setItem('sivacad_user', JSON.stringify(usuario))
    F->>U: Redirigir según rol (/app/dashboard o /app/alumno)
Endpoints involucrados: POST /api/auth/login · GET /api/auth/me

3. Registro de Usuario

POST /auth/register · Transacción: INSERT usuarios + alumnos/docentes + kardex
El formulario de registro captura nombres, apellidos, correo, contraseña, rol y datos específicos (matrícula o número de empleado). El backend valida el dominio del correo, verifica que no exista un usuario con el mismo correo, capitaliza nombres, hace hash a la contraseña con bcrypt (12 rounds), e inicia una transacción. Inserta en `usuarios`, y según el rol inserta en `alumnos` (con `kardex_alumno`) o `docentes`. Finalmente, firma un token JWT y devuelve los datos del nuevo usuario. 
sequenceDiagram
    actor U as Usuario
    participant F as Frontend (RegisterPage)
    participant B as Backend (auth.js: register)
    participant DB as MySQL

    U->>F: Completa formulario de registro
    F->>B: POST /api/auth/register { nombres, apellidos, correo, contrasena, rol, ... }
    B->>B: Validar correo institucional
    B->>DB: SELECT id_usuario WHERE correo_institucional = ? (evitar duplicado)
    DB-->>B: Usuario no existe
    B->>B: bcrypt.hash(contrasena, 12)
    B->>DB: BEGIN TRANSACTION
    B->>DB: INSERT INTO usuarios (nombres, apellidos, correo, hash, 'Activo', id_rol)
    DB-->>B: id_usuario
    alt Rol = alumno
        B->>DB: INSERT INTO alumnos (id_usuario, matricula, curp, id_carrera, ...)
        DB-->>B: id_alumno
        B->>DB: INSERT INTO kardex_alumno (id_alumno, numero_control, qr_token, ...)
    else Rol = docente / coordinador / administrador / soporte
        B->>DB: INSERT INTO docentes (id_usuario, numero_empleado, especialidad, ...)
    end
    B->>DB: COMMIT
    B->>B: jwt.sign({ id_usuario, correo, rol }, JWT_SECRET)
    B-->>F: 201 { ok, token, usuario: { nombres, correo, rol, ... } }
    F->>F: Guardar token y usuario en localStorage
    F->>U: Redirigir al dashboard correspondiente
Endpoints involucrados: POST /api/auth/register

4. Recuperación de Contraseña

POST /auth/forgot-password · Rate limit: 5 req/15min · Nodemailer + Gemini no involucrado
El usuario ingresa su correo institucional. El frontend valida el dominio y envía la solicitud. El backend valida el correo nuevamente (con soporte de subdominios), busca al usuario en la BD, verifica si está activo, y si todo es correcto, genera un token criptográfico de 64 caracteres hex (crypto.randomBytes(32)). Inicia una transacción, marca como usados los tokens previos del usuario, inserta el nuevo token con expiración de 15 minutos, y envía el correo via SMTP (Nodemailer) con un enlace al frontend. Si no hay SMTP configurado o falla, en development muestra el enlace en consola y en la UI. Los mensajes de error son genéricos para evitar enumeración de correos. 
sequenceDiagram
    actor U as Usuario
    participant F as Frontend (ForgotPasswordPage)
    participant B as Backend (auth.js: forgotPassword)
    participant DB as MySQL
    participant SMTP as Nodemailer

    U->>F: Ingresa correo institucional
    F->>F: isInstitutionalEmail() + normalizar
    F->>B: POST /api/auth/forgot-password { correo }
    B->>B: Rate limit check (5 solicitudes / 15 min)
    B->>B: validateInstitutionalEmail (soporte subdominios)
    B->>DB: SELECT usuario WHERE correo_institucional = ?
    DB-->>B: Usuario encontrado
    B->>B: usuario.estado === 'Activo'?
    B->>B: crypto.randomBytes(32).toString('hex') = token 64 chars
    B->>B: expiresAt = Date.now() + 15*60*1000
    B->>DB: BEGIN TRANSACTION
    B->>DB: UPDATE password_resets SET used=1 WHERE id_usuario=? AND used=0 (cleanup)
    B->>DB: INSERT INTO password_resets (id_usuario, token, expires_at, created_at)
    B->>DB: COMMIT
    B->>B: resetUrl = FRONTEND_URL + RESET_PASSWORD_PATH + token
    B->>SMTP: sendPasswordResetEmail({ to, name, resetUrl })
    alt SMTP configurado y funciona
        SMTP-->>B: Correo enviado exitosamente
        B-->>F: 200 { ok, message: "Se envió un enlace..." }
    else SMTP placeholder o falla (development)
        B-->>F: 200 { ok, message: "...", devMode: true, resetUrl }
        F-->>U: Muestra el enlace en la UI (modo local)
    else SMTP falla (production)
        B-->>F: 503 { ok: false, message: "No fue posible enviar..." }
    end
Endpoints involucrados: POST /api/auth/forgot-password · Tablas: usuarios, password_resets

5. Restablecer Contraseña

POST /auth/reset-password/:token · Validación: SQL expires_at > NOW() + código
El usuario recibe el enlace de recuperación (por correo o en UI en dev) y es redirigido a ResetPasswordPage con el token en la URL. El frontend valida que la nueva contraseña tenga al menos 8 caracteres y que coincida con la confirmación. El backend busca el token en la tabla password_resets con condición SQL used=0 AND expires_at > NOW(). Si no existe o expiró, se rechaza. Si todo es válido, se hace hash a la nueva contraseña con bcrypt, se actualiza el registro del usuario, se marca el token como usado y se devuelve éxito. El frontend redirige a /login tras 1.4 segundos. 
sequenceDiagram
    actor U as Usuario
    participant F as Frontend (ResetPasswordPage)
    participant B as Backend (auth.js: resetPassword)
    participant DB as MySQL

    U->>F: Click en enlace de recuperación (token en URL)
    F->>F: Extraer token de useParams()
    F-->>U: Formulario: nueva contraseña + confirmar
    U->>F: Ingresa nueva contraseña (>=8 caracteres, coinciden)
    F->>B: POST /api/auth/reset-password/:token { contrasena }
    B->>DB: SELECT pr.*, u.estado FROM password_resets pr JOIN usuarios u WHERE pr.token=? AND pr.used=0 AND pr.expires_at>NOW()
    DB-->>B: resetRow (válido) o null
    alt Token válido no expirado
        B->>B: bcrypt.hash(contrasena, 12)
        B->>DB: BEGIN TRANSACTION
        B->>DB: UPDATE usuarios SET contrasena_hash=? WHERE id_usuario=?
        B->>DB: UPDATE password_resets SET used=1 WHERE id_reseteo=?
        B->>DB: COMMIT
        B-->>F: 200 { ok, message: "Contraseña actualizada" }
        F->>F: setTimeout() 1.4s
        F->>U: Redirigir a /login
    else Token inválido o expirado
        B-->>F: 400 { ok: false, message: "Enlace no válido o ha expirado" }
        F-->>U: Muestra error
    end
Endpoints involucrados: POST /api/auth/reset-password/:token · Tablas: password_resets, usuarios

9. OCR de Actas

Ruta: actasOCR.js (1127 lines) · Multer + Gemini API + PDF/imágenes
Este módulo procesa actas de calificaciones escaneadas (PDF, JPG, PNG) utilizando la API de Google Gemini para extraer texto e identificar calificaciones, materias, alumnos y docentes. El proceso incluye subida de archivo con multer, conversión de PDF a imagen base64 si es necesario, llamada a Gemini con prompt especializado, parsing de la respuesta JSON, y almacenamiento de datos en tablas temporales. La validación final confirma e importa los datos a las tablas de calificaciones.

11. IA de Becas

Ruta: iaBecas.js (747 lines) · Servicios: BecasEdomexService, becasVectorStore, becasTools · Tablas: becas_fuentes, becas_chunks
El módulo de becas ofrece un catálogo de 14 becas oficiales (SECTI Edoméx, COMECYT, Santander, BBVA, etc.) con búsqueda semántica usando embeddings simples (basados en conteo de palabras, no es embedding real de ML). Los usuarios pueden preguntar sobre becas, el sistema clasifica la intención, busca en el vector store, rankea resultados y responde. También se puede calcular elegibilidad de un alumno comparando su promedio y semestre contra los requisitos de cada beca. El catálogo se puede "ingestar" (poblar/actualizar) desde fuentes oficiales.

18. Arquitectura de Rutas del Backend

routes/index.js · mountIfAvailable(pattern) · SafeRequire + auto-detección de módulos
El backend usa un sistema de montaje dinámico: app.js monta routes/index.js en /api, y el index usa mountIfAvailable(routePath, relativeModulePath) que intenta requerir el archivo del módulo; si existe, lo monta en la ruta especificada. Esto permite que los módulos se habiliten/deshabiliten simplemente agregando o quitando archivos. Cada sub-router tiene sus propios middlewares (JWT + roles) y sus propios controladores. Express encadena los routers en orden de registro; si dos routers comparten prefijo (como los 4 de /ia/desercion), Express evalúa todos hasta encontrar una ruta que coincida.

19. Arquitectura del Frontend — AuthContext, Roles y API Request

AuthContext.jsx · ProtectedRoute · api.js · request() · emitAuthError · localStorage
El frontend gestiona la autenticación mediante un AuthContext que almacena el usuario y el token JWT en el estado de React y en localStorage. El ProtectedRoute verifica si hay sesión activa antes de renderizar cualquier página, y si no hay, redirige al login. El api.js centraliza todas las llamadas HTTP mediante una función request() que inyecta automáticamente el token Bearer, maneja timeouts, errores de red, y dispara un evento personalizado sivacad:auth-error en 401 para que AuthContext limpie la sesión. Los roles se verifican tanto en frontend (componentes canAccess*) como en backend (middleware role()). Existe un bug crítico corregido donde las páginas que sólo destructureaban { user } de useAuth() perdían el token porque el objeto usuario ya no contiene el token; se corrigió a { user, token }. También se agregó una migración en AuthContext para recuperar tokens almacenados en versiones previas dentro del objeto usuario.
