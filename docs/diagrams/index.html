1. Flujo Global del Sistema

Arquitectura general: Frontend → API REST → MySQL → Servicios externos (Gemini, SMTP, Flask, PHP)
El sistema sigue una arquitectura de tres capas (presentación, lógica de negocio, datos). El Frontend en React/Vite se comunica con el Backend en Express.js mediante peticiones HTTP con formato JSON. El backend procesa la lógica de negocio, consulta la base de datos MySQL y, cuando es necesario, llama a servicios externos como Gemini API (OCR, chatbot, becas, bienestar), SMTP (correos), Flask ML (predicción de deserción/bienestar) y PHP CLI (generación de PDF/Excel). El middleware JWT protege las rutas privadas y el middleware de roles (RBAC) controla los permisos por perfil.

2. Autenticación — Login

POST /api/auth/login · Middleware: validateInstitutionalEmail · JWT + bcrypt
El usuario ingresa su correo y contraseña en el formulario de Login. Se normaliza el email (minúsculas, trim) y se envía al backend. El middleware verifica que el dominio del correo esté en la lista blanca institucional. El controlador busca el usuario en la BD, verifica que esté Activo, compara la contraseña con bcrypt (12 rounds), y si es válida, genera un token JWT firmado con el secret del .env. Se devuelve el token y los datos del usuario. El frontend almacena el token y los datos en localStorage y redirige según el rol.

    alt Contraseña incorrecta

        B-->>F: 200 { ok: true, token, usuario }

        F->>U: Redirigir según rol
    end

3. Registro de Usuario

POST /auth/register · Transacción: INSERT usuarios + alumnos/docentes + kardex
El formulario de registro captura nombres, apellidos, correo, contraseña, rol y datos específicos (matrícula o número de empleado). El backend valida el dominio del correo, verifica que no exista un usuario con el mismo correo, capitaliza nombres, hace hash a la contraseña con bcrypt (12 rounds), e inicia una transacción. Inserta en usuarios, y según el rol inserta en alumnos (con kardex_alumno) o docentes.

    B->>DB: BEGIN TRANSACTION
    B->>DB: INSERT INTO usuarios (nombres, apellidos, correo, hash, 'Activo', id_rol)

    F->>U: Redirigir al dashboard correspondiente

4. Recuperación de Contraseña

POST /auth/forgot-password · Rate limit: 5 req/15min · Nodemailer + crypto
El usuario ingresa su correo institucional. El backend valida el dominio, busca al usuario en la BD, verifica si está activo, y genera un token criptográfico de 64 caracteres hex (crypto.randomBytes(32)). Inicia una transacción, marca como usados los tokens previos, inserta el nuevo token con expiración de 15 minutos, y envía el correo vía SMTP con un enlace al frontend. Los mensajes de error son genéricos para evitar enumeración de correos.
sequenceDiagram
    actor U as Usuario
    participant F as Frontend
    participant B as Backend
    participant DB as MySQL
    participant SMTP as Nodemailer
    U->>F: Ingresa correo institucional
    F->>F: isInstitutionalEmail()
    F->>B: POST /api/auth/forgot-password { correo }
    B->>B: Rate limit check
    B->>B: validateInstitutionalEmail
    B->>DB: SELECT WHERE correo_institucional = ?
    DB-->>B: Usuario encontrado y activo
    B->>B: crypto.randomBytes(32) = token 64 chars
    B->>DB: BEGIN TRANSACTION
    B->>DB: UPDATE password_resets SET used=1 (cleanup)
    B->>DB: INSERT INTO password_resets (token, expires_at)
    B->>DB: COMMIT
    B->>SMTP: sendPasswordResetEmail()
    alt SMTP funciona
        B-->>F: 200 { ok, message }
    else Development mode
        B-->>F: 200 { ok, devMode: true, resetUrl }
    end
Endpoints: POST /api/auth/forgot-password · Tablas: usuarios, password_resets

5. Restablecer Contraseña

POST /auth/reset-password/:token · Validación: SQL expires_at > NOW() + used=0
El usuario recibe el enlace de recuperación y es redirigido a ResetPasswordPage con el token en la URL. El backend busca el token en la tabla password_resets con condición used=0 AND expires_at > NOW(). Si no existe o expiró, se rechaza. Si es válido, se hace hash a la nueva contraseña con bcrypt, se actualiza el usuario, se marca el token como usado.
sequenceDiagram
    actor U as Usuario
    participant F as Frontend
    participant B as Backend
    participant DB as MySQL
    U->>F: Click en enlace (token en URL)
    F-->>U: Formulario: nueva contraseña
    U->>F: Ingresa contraseña (>=8 chars)
    F->>B: POST /api/auth/reset-password/:token { contrasena }
    B->>DB: SELECT pr.*, u.estado FROM password_resets pr JOIN usuarios u WHERE pr.token=? AND pr.used=0 AND pr.expires_at>NOW()
    DB-->>B: resetRow válido o null
    alt Token válido
        B->>B: bcrypt.hash(contrasena, 12)
        B->>DB: TRANSACTION: UPDATE usuarios + UPDATE password_resets SET used=1
        B->>DB: COMMIT
        B-->>F: 200 { ok, message }
        F->>U: Redirigir a /login
    else Inválido o expirado
        B-->>F: 400 { ok: false, message }
        F-->>U: Muestra error
    end
Endpoints: POST /api/auth/reset-password/:token

19. Arquitectura del Frontend — Auth, Roles y API Request

AuthContext.jsx · ProtectedRoute · api.js · request() · emitAuthError
El frontend gestiona la autenticación mediante AuthContext que almacena usuario y token JWT en React state y localStorage. ProtectedRoute verifica sesión activa antes de renderizar. api.js centraliza las llamadas HTTP, inyecta token Bearer, maneja timeouts y dispara evento sivacad:auth-error en 401.
